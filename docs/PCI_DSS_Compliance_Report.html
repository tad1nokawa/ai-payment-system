1 ネットワークセキュリティコントロールの導入・維持

PCI DSS v4.0 要件概要: ファイアウォール、セキュリティグループ、VPC等のネットワークセキュリティコントロールを導入し、CDEを保護する。v4.0では「ファイアウォール」から「ネットワークセキュリティコントロール (NSC)」に名称が拡大され、クラウド環境のセキュリティグループも明確に対象となった。

サブ要件現在の設計状況評価対応方針
1.2 NSCの構成標準の定義 VPC Peering / PrivateLinkの方針記載あり。セキュリティグループの詳細ルールは未定義 ⚠️ 部分適合 NSC構成標準書を別途作成
1.3 CDEへの不正アクセス制限 Zone A「完全隔離VPC, WAF, IDS/IPS」と記載。Zone B「通常のVPC, WAF」と記載 ✅ 適合 実装時にSG/NACLルールの詳細設計
1.4 信頼できるネットワークとの境界制御 ALB + CloudFront構成が設計済み。WAF配置方針あり ✅ 適合 WAFルールの詳細設計
1.5 CDEと外部ネットワークの分離 Zone A = CDE専用VPC、Zone B = 管理系VPC。VPC Peering/PrivateLinkで接続 ✅ 適合 PrivateLink構成の詳細化
✅ 総合評価: 設計適合 Zone A / Zone Bの2VPC分離構成はPCI DSS要件1のネットワーク分離要件を満たす。VPC Peering/PrivateLinkによる環境間接続も適切な方針。実装時にセキュリティグループの詳細ルール（ポート/プロトコル/CIDR）を定義する必要あり。

3 保存データ（アカウントデータ）の保護

PCI DSS v4.0 要件概要: 保存されたカードデータの暗号化・トークン化・マスキング。ディスク/パーティションレベルの暗号化のみでは不十分（v4.0新要件）。暗号鍵管理、PANハッシュの暗号化キー付きハッシュ化が必須。

サブ要件現在の設計状況評価対応方針
3.1 データ保持の最小化 Zone Bはカード番号非保持。トークン参照のみ。customer_cards.card_token/subscription_users.card_tokenはCDE側トークン ✅ 適合 Zone A側のデータ保持ポリシーを江成チームに確認
3.2 SAD（認証データ）非保存 CVVはCDE内のみ（Zone A）。Zone BにCVV保存カラムなし ✅ 適合 CVVの認可後即時削除を確認
3.3 PAN表示のマスキング 管理画面上のカード番号表示設計あり（M03取引詳細:下4桁のみ表示） ✅ 適合 全画面のカード番号表示箇所をレビュー
3.4 PAN保存時の暗号化 Zone AにHSM/KMS記載あり。Zone Bはトークンのみ保存 ✅ 適合 HSM仕様（FIPS 140-2 Level 3以上）を確認
3.5 PANの暗号化 — ディスクレベル暗号化の禁止（v4.0新） HSM/KMSによるアプリケーションレベル暗号化の方針 ⚠️ 要確認 Zone AでRDS暗号化+アプリレベル暗号化の二重保護を確認
3.6/3.7 暗号鍵管理 HSM/KMS記載あり。鍵ローテーション・デュアルコントロールの詳細は未定義 ⚠️ 部分適合 鍵管理ポリシー策定（年次ローテーション、split knowledge等）
✅ 総合評価: 設計適合 Zone B（管理系）はカード番号非保持設計で、PCI DSS要件3の大部分がスコープ外。Zone A（CDE）のHSM/KMS方針は適切。トークン化によるスコープ縮小が最大の強み。
💡 v4.0 重要変更点 PCI DSS v4.0では、ディスク/パーティションレベルの暗号化のみでは要件3.4を満たさない。リムーバブルメディア以外では、アプリケーションレベルの暗号化（カラムレベル暗号化等）が必須。Zone A（CDE）でのDB暗号化方式を確認すること。
また、PANのハッシュ保存には暗号化キー付きハッシュ（HMAC等）が必須となった（v4.0 Req 3.5.1.1）。

4 カードデータ通信の暗号化

PCI DSS v4.0 要件概要: オープンなパブリックネットワークでのカードデータ送受信時に強力な暗号化を使用する。TLS 1.2以上必須。

サブ要件現在の設計状況評価対応方針
4.1 暗号化プロトコル定義 通信経路表にHTTPS/WSS/mTLSが明記。TLSバージョンの明記なし ⚠️ 部分適合 TLS 1.2以上を明記、TLS 1.0/1.1の明示的無効化
4.2 PANの非暗号化送信禁止 P02: CDE iframe内でカード入力→postMessageでtoken返却。PANがZone B側に流れない設計 ✅ 適合 —
4.2.1 証明書管理 ドメイン構成設計あり（pay/admin/apply/cde.aipayment.jp）。証明書管理方針は未記載 ⚠️ 部分適合 ACM証明書の自動更新 + 証明書棚卸しプロセス策定
✅ 総合評価: 設計適合 iframeトークン化によりPANがZone Bに流れない設計は適切。Zone A↔B間のmTLS方針も記載済み。実装時にTLSバージョン/暗号スイートの詳細設定を確定する。

PCI DSS v4.0 要件概要: 全システムをマルウェアから保護。v4.0新要件: フィッシング攻撃への対策（5.4.1）が必須化。

サブ要件現在の設計状況評価対応方針
5.2 マルウェア対策ソリューション 設計文書にマルウェア対策の記載なし。ECS Fargate（コンテナ）はエージェント型が困難 ⚠️ 部分適合 AWS GuardDuty + ECRイメージスキャン + コンテナランタイムセキュリティ
5.3 マルウェア定期スキャン CloudWatch監視の記載あり。スキャン方針は未定義 ⚠️ 部分適合 ECRイメージスキャン + S3アップロードファイルスキャン
5.4.1 フィッシング対策（v4.0新） 設計文書に記載なし 🔴 未定義 DMARC/SPF/DKIM設定 + 社内フィッシング訓練プログラム策定
⚠️ 総合評価: 部分適合 クラウドネイティブ（ECS Fargate）環境では従来のアンチウイルスエージェントが使えないため、GuardDuty/ECRスキャン/コンテナセキュリティでの代替が必要。フィッシング対策はv4.0新要件で組織的な対応が必要。

6 セキュアなシステム・アプリケーションの開発と維持

PCI DSS v4.0 要件概要: セキュリティパッチの適用、セキュアコーディング、Webアプリケーションの保護。v4.0新要件: 決済ページのスクリプト管理（6.4.3）、ソフトウェアインベントリ（6.3.2）が必須化。

サブ要件現在の設計状況評価対応方針
6.1 セキュリティ脆弱性管理プロセス 設計文書に脆弱性管理プロセスの記載なし 🔴 未定義 脆弱性管理ポリシー策定（クリティカル30日以内パッチ適用等）
6.2 セキュアな開発プロセス React 19 + Vite 6構成。セキュアコーディング基準の記載なし ⚠️ 部分適合 OWASP Top 10ベースのセキュアコーディングガイドライン策定
6.3.2 ソフトウェアインベントリ（v4.0新） package.json/package-lock.jsonで依存管理。正式インベントリなし ⚠️ 部分適合 SBOM（Software Bill of Materials）生成プロセス導入
6.4.1 WAF導入 Zone A/B共にWAF記載あり ✅ 適合 WAFルール詳細設計
6.4.3 決済ページスクリプト管理（v4.0新・重要） P02はCDE iframe設計。メインページのスクリプト管理方針は未定義 ⚠️ 要対応 CSP（Content Security Policy）+ SRI（Subresource Integrity）+ スクリプトインベントリ
🔴 重要: 要件6.4.3 — 決済ページのスクリプト管理（v4.0新要件） PCI DSS v4.0で最も注目されている新要件の一つ。決済ページ（P02）で実行される全JavaScriptスクリプトについて：
① インベントリ管理: 全スクリプトのURL/ハッシュ値/目的を文書化
② 承認プロセス: 各スクリプトの必要性を正式に承認
③ 整合性検証: SRI (Subresource Integrity) によるハッシュ検証

現在の設計の強み: CDE iframe内でカード入力する設計のため、P02メインページのJSはカードデータに直接触れない。ただし、iframe外のスクリプトもスキミング攻撃の対象となるため、CSP + SRIの実装は必須。

要件11.6.1との連携: 決済ページのHTTPヘッダーやスクリプトの不正変更を検知するメカニズム（週次以上のチェック）も必須。

8 ユーザー識別と認証管理

PCI DSS v4.0 要件概要: 一意のユーザーID、強力なパスワード（12文字以上）、MFA（全CDEアクセスに必須）。v4.0で大幅強化。

サブ要件現在の設計状況評価対応方針
8.2 一意のユーザーID M12でユーザー個別ID管理。共有アカウント禁止の方針 ✅ 適合 —
8.3.6 パスワード要件（v4.0: 12文字以上） 現在の設計: 8文字以上/英数字混合 → PCI DSS v4.0非適合 🔴 非適合 12文字以上 + 英大小文字+数字+記号必須に引き上げ
8.3.7 パスワード履歴（過去4回不可） 設計文書に記載あり（AQUAGATES参考） ✅ 適合 —
8.3.4 ロックアウト（10回以上で一時ロック） 設計文書: 6回失敗でロック（30分） → PCI DSS: 10回以内なのでOK ✅ 適合 —
8.4 MFA — 全CDEアクセスに必須（v4.0新） M13セキュリティ設定にMFA設定UIあり。CDE管理アクセスのMFA方針は未明記 ⚠️ 部分適合 全管理画面アクセス + CDE管理アクセスにMFA必須化
8.5 MFAシステム構成 MFAの具体的な技術実装（TOTP/WebAuthn等）は未定義 ⚠️ 部分適合 TOTP + WebAuthn（FIDO2）の二択提供を推奨
8.6 システム/サービスアカウント管理 Zone A↔B間の内部API認証（mTLS/IAM/API Key）は方針のみ ⚠️ 部分適合 サービスアカウント一覧 + 認証方式の確定
🔴 要対応: パスワードポリシーがv4.0非適合 現在の「8文字以上/英数字混合」はPCI DSS v4.0の最低要件「12文字以上」を満たしていない。
推奨ポリシー:
• 文字数: 12〜32文字
• 複雑性: 英大文字 + 英小文字 + 数字 + 記号（各1文字以上）
• 履歴: 過去4回と同一不可
• 有効期限: 90日
• ロックアウト: 6回失敗で30分ロック（現行設計のまま — PCI DSS要件の10回以内より厳格でOK）
• MFA: 全ユーザー必須（CDEアクセスに限らず推奨）

10 ログの記録と監視

PCI DSS v4.0 要件概要: 全システムコンポーネントとカードデータへのアクセスを記録・監視する。v4.0新要件: 自動化された監査ログレビュー（10.4.1.1）が必須化。手動レビューのみでは不適合。

サブ要件現在の設計状況評価対応方針
10.1 監査ログの生成 audit_logs テーブル設計済み。操作ログAPI（M13）実装。90日保持、月次パーティション ✅ 適合 —
10.2 ログ記録項目 audit_logsに操作者/操作内容/タイムスタンプ記録設計 ✅ 適合 ログ記録項目の詳細定義（IP, User-Agent等の追加）
10.3 ログの改竄防止 設計文書に改竄防止策の明記なし ⚠️ 部分適合 CloudWatch Logs + S3（Write-Once）+ ログ完全性モニタリング
10.4.1.1 自動化されたログレビュー（v4.0新） CloudWatch監視の記載あり。SIEM/自動レビューの方針は未定義 🔴 未定義 SIEM導入（AWS Security Hub + CloudWatch Logs Insights or 3rd party SIEM）
10.5 監査ログの保管（12ヶ月、3ヶ月即時アクセス） 現在の設計: 90日保持 → PCI DSS: 12ヶ月保持（3ヶ月即時アクセス）が必要 🔴 非適合 ログ保持期間を12ヶ月に延長。直近3ヶ月=即時アクセス、残り9ヶ月=S3 Glacier等で保管
🔴 重要: ログ保持期間の延長が必要 現在の設計は「90日保持」だが、PCI DSS v4.0 要件10.5.1では最低12ヶ月保持（直近3ヶ月は即時検索可能）を要求。

推奨アーキテクチャ:
• 直近3ヶ月: RDS audit_logs + CloudWatch Logs（即時検索可能）
• 3〜12ヶ月: S3 Standard（アーカイブだが即日アクセス可能）
• 12ヶ月以降: S3 Glacier Deep Archive（コンプライアンス保管）

v4.0新要件（10.4.1.1）: 手動のログレビューだけでは不適合。SIEM等の自動化ツールによる日次レビュー・異常検知アラートが必須。
